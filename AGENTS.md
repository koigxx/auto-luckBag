## 核心业务规则

### 福袋类型

- 支持全部类型、实物福袋、钻石福袋、其他类型。
- 用户可任意组合，例如同时勾选实物和钻石。

### 钻石预算

- 所有直播间共享一个钻石预算总额。
- 已使用钻石跨直播间累计。
- 预算不足时跳过需要灯牌或粉丝团付费的福袋，但不影响不需要付费粉丝团任务的福袋。
- 自动验证直播间后，如果识别到该福袋需要加入粉丝团或点亮灯牌，必须先检查有效钻石预算：
  - 有预算时，才允许直接停留或加入主候选池；
  - 预算达到上限时，跳过该直播间；
  - 不需要粉丝团或灯牌的福袋不受钻石预算限制。
- 支持“允许使用利润”配置：
  - 关闭时，有效预算 = `diamondBudget`；
  - 开启时，有效预算 = `diamondBudget + diamondWonAmount`；
  - `diamondWonAmount` 来自钻石中奖结果文本解析，解析不到数量时按 0 处理。

### 自动任务

- 要求关注主播时自动点击关注，始终启用，不提供关闭项。
- 要求评论时尝试识别口令并自动发送评论。
- 要求加入粉丝团或点亮灯牌时，先检查有效钻石预算，预算足够才执行。
- 当同一个福袋同时要求评论和加入粉丝团时，流程应优先完成评论；评论后如果福袋弹窗仍在且按钮变为“加入粉丝团(1钻/1钻石)”，应直接在当前弹窗继续点击粉丝团按钮，不要重复点击福袋入口。
- 当福袋要求加入粉丝团时，不需要单独先点关注主播；抖音的粉丝团加入流程会自动处理关注。
- 粉丝团点击必须限定在福袋弹窗中的明确按钮，例如 `加入粉丝团(1钻石)`、`加入粉丝团（1钻)`、`点亮粉丝团（1钻)`。
- 禁止把主播头像旁边的 `+加粉丝团` 入口、整块福袋弹窗父容器、左侧礼物流里的 `送点亮粉丝团` 文本当作粉丝团按钮。

### 多房间

- 每个正式监控的直播间使用一个 Playwright Page。
- 所有 Page 共享同一个 BrowserContext，复用登录态。
- 建议最多同时监控 5 个直播间。
- 点击停止时必须停止扫描、验证、直播间监控和所有页面操作，并关闭已监控房间。
- 福袋开奖等待结束后，应移除监控并关闭对应页面，避免 tab 持续增加。

### 福袋检测

1. WebSocket 帧拦截：解析直播间推送消息识别福袋。
2. Network Response 被动监听：只解析当前页面自然加载产生的响应，不主动请求后台接口。主识别目标是 `/webcast/lottery/` 或包含 `lottery_info` 的福袋结构化响应。
3. DOM MutationObserver 兜底：只认可见、尺寸正常、位于视口内的福袋入口，隐藏元素不能触发点击。
4. ShortTouch 倒计时：真实福袋入口常出现在 `#ShortTouchLayout`，其中 `.ycjwPFJI` 的 `mm:ss` 文本可作为可见 DOM 倒计时。
5. 页面文本只作为辅助信号，不作为主候选池的可靠倒计时来源。

### 开奖倒计时

- 获取福袋开奖倒计时是主流程的硬约束。
- 验证通过但没有可靠倒计时的直播间不得进入主候选池。
- 候选池和仪表盘的倒计时应优先使用绝对开奖时间 `drawAt` 计算，即 `drawAt - Date.now()`。
- 从结构化响应中解析到 `draw_time/current_time` 时，应按服务端时间和本机时间偏移校准本地 `drawAt`，避免 UI 倒计时和真实福袋倒计时相差数秒。
- 渲染端不应自行每秒递减倒计时，倒计时以主进程推送为准。
- 主候选池只接收：
  - `countdownSource === "network"` 且 `countdownConfidence === "exact"`，即来自结构化福袋响应；
  - 或 `countdownSource === "visible-dom"`，即来自可见 ShortTouch 倒计时。
- 普通文本倒计时、未知来源倒计时、其他接口里的泛 countdown 都不应驱动正式进房。
- 有福袋迹象但缺少可靠倒计时的房间进入低频复验队列。

### 自动运行流程

自动运行以低频串行方式执行，目标是降低抖音风控风险。

1. 扫描发现候选：
   - 访问抖音直播入口页。
   - 从 DOM 和页面自然响应中提取直播间链接。
   - 过滤非直播路径、保留字路径、UUID 路径和明显无效路径。
   - 不在扫描阶段批量进入直播间。

2. 逐个进房验证：
   - 每轮最多验证 1 个候选。
   - 验证进房串行执行，使用导航队列控制间隔。
   - 进入候选直播间后停留约 20-32 秒，多轮采样，避免网络慢或内容延迟加载导致漏判。
   - 通过 WebSocket、结构化福袋响应、可见 ShortTouch 倒计时、可见 DOM 入口和任务文本综合评分。
   - 验证失败的候选冷却 30 分钟，避免反复进入同一个无效直播间。

3. 候选池等待：
   - 只有拿到可靠倒计时的房间才进入候选池。
   - 候选池按剩余开奖时间排序。
   - 候选池上限可配置，默认 5，最大 5。
   - 剩余时间小于等于 `enterBeforeSeconds` 时才正式进房参与。
   - 加入候选池前必须检查与现有候选和正在监控房间的开奖时间冲突：开奖时间间隔小于 `enterBeforeSeconds` 时跳过，减少同一时间只能处理一个直播间导致的冲突。

4. 验证时临近开奖：
   - 如果验证时发现剩余时间位于 `(enterBeforeSeconds - 20, enterBeforeSeconds + 15]` 秒，不再记录到候选池。
   - 直接接管当前验证页参与福袋，并等待开奖后继续后续队列。
   - 如果验证时剩余时间已经小于等于 `enterBeforeSeconds - 20` 秒，视为太晚，不直接停留，避免任务完成时间不足。
   - 直接停留前也必须执行预算检查和开奖时间冲突检查。

### 重复日志与页面关闭

- DOM 检测只负责触发处理，不应每几秒重复打印“检测到可见福袋入口”。
- 参与成功后，该房间进入等待开奖冷却，期间不重复点击同一个福袋。
- 如果参与成功时拿到了剩余秒数，RoomManager 会安排开奖后自动关闭该直播间页面。
- AutoRunner 也会按候选倒计时安排开奖后移除房间，二者互为兜底。

### 风控策略

- 默认扫描/验证节奏偏慢，优先稳定。
- 有待验证队列时不重复扫描入口页。
- 验证直播间串行执行，禁止并发快速切换。
- 正式进房也串行执行。
- 点击操作添加短随机延迟。
- 只被动监听当前页面自然产生的 WebSocket 和 Network Response，不主动请求后台接口。
- 遇到 95152、安全验证、滑块、访问频繁等风控信号时暂停扫描和验证 10 分钟。
- 用户停止后立即关闭发现页、验证页和已监控房间。
